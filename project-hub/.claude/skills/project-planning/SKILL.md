# Project Planning

I code manually. Planning output = a plan I execute myself, not code. Keep plans short enough to fit on one screen.

## Plan format (always this shape)

1. **Goal** — one sentence, what "done today" means
2. **Learning target** — the ONE thing this project teaches me
3. **Core loop** — the 3–5 steps that make it minimally work end-to-end
4. **Stretch** — 2–3 nice-to-haves, explicitly labeled cuttable
5. **First 30 minutes** — the exact first tasks, so I start instead of stalling

## Scoping rules

- MVP = walking skeleton first: thinnest end-to-end path (input → logic → output), ugly is fine. Features attach to a working skeleton
- Anything not reachable in ~4–6 focused hours goes to Stretch or a future project
- One new technology per project max; everything else uses stacks I know
- Mid-build "should I also add X?" → default answer NO; add to Stretch, finish the core. Only override if X blocks the core

## When I bring a vague idea

Ask at most 2 clarifying questions (who uses it, what's the single core action), then produce the plan. Don't interrogate — assume sensible defaults and state them.

## Definition of done (every project)

Works end-to-end · a few tests pass · README with learning target · committed in small chunks · added to project-atlas data file
